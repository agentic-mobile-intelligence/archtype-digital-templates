# Chapter 11: Template Testing

A template is a promise. When someone forks it, they expect the scaffolding to work on the first run — builds pass, tests pass, the example command emits the example output. That promise is easy to make on the day you publish and surprisingly hard to keep six months later, after upstream dependencies have shifted, after a contributor refactored the build script, after a downstream engineer quietly fixed a path and never backported the fix. Testing is how the promise survives.

The hard part is that a template is not one artifact. It is three: the archetype repository itself, the instances that have been forked from it, and the sync boundary between them. Each has its own failure mode, and each needs its own tests. A template whose CI goes green tells you only that the archetype compiles. It does not tell you that a fresh fork compiles, that last month's downstream instances still run, or that your most recent upstream change did not silently break every consumer. This chapter walks through the three layers of testing that, together, let you ship template changes with confidence.

## Testing the Template Itself

The first and most neglected category of test is the one that runs against the archetype in isolation. It is neglected because it feels redundant — of course the archetype works; we would notice if it did not. But "we would notice" is rarely true for a template, because the archetype's code paths are mostly example code. Nobody is exercising them in production. If the example `npm run build` silently regresses because an exported symbol was renamed, the archetype's own CI may still pass while every downstream fork breaks on day one.

Template-level tests should cover three things. First, the scaffolding itself has to execute: whatever entry points the template advertises — `make dev`, `npm run test`, `pytest`, `cargo build` — need to be invoked by CI and required to pass. This is the template's own smoke test, and it should run on every PR to the archetype. Do not treat these as optional or informational. A red build on the archetype means a broken promise to every future consumer.

Second, the template needs fork-simulation tests. The most realistic way to know whether a newly forked copy will work is to fork it, in CI, and run the setup steps a new consumer would run. A typical fork-simulation job copies the archetype into a scratch directory (stripping archetype-only files like `.github/workflows/archetype-*.yml`, the `ARCHETYPE.md`, and anything else that lives only upstream), runs whatever initialization the README prescribes, and then executes the smoke suite. If the README says `cp .env.example .env && npm install && npm test`, CI runs exactly that. A green job tells you a new fork, today, will work. A red one tells you before someone files an issue.

Third, template-level tests should cover the template's own contracts. If your `publishing/` directory is promised to be the interface consumers rely on, then removing a file from that directory is a breaking change, and a test should fail when it happens. The same goes for any documented file path, script name, or environment variable. These contract tests are cheap — usually a shell script that greps for required files and required function names — but they catch the slow, silent breakage that accumulates when nobody is looking. Think of contract tests as the template equivalent of an API's backwards-compatibility check: they encode the promises that versioning alone cannot enforce.

## Smoke Tests for Downstream Instances

The second category is tests that run inside instances. These exist because the archetype cannot know, on its own, whether a downstream fork still works after customization. An instance has modified content, local configuration, and usually some production data shape that the archetype has never seen. A test suite that runs in the instance and exercises the instance's real flows is the only thing that tells you whether the template's promises have survived contact with real-world usage.

The goal for a downstream smoke test is narrow: prove that the instance, as customized, still boots, still builds, still passes whatever the critical happy-path checks are. This is not a full integration suite — integration tests are a separate concern that each instance owns independently. A smoke test answers a simpler question: is this thing fundamentally broken. That usually means a startup check, one or two end-to-end path exercises, and a sanity check on any tooling provided by the template (the drift-check script, the backport script, the publishing pipeline).

The instance's smoke tests should be inherited from the template, not rewritten. The archetype ships a canonical smoke test under `publishing/smoke/` or similar, and instances run it unchanged as part of their own CI. When the template improves its smoke tests, the improvement propagates through the normal sync pattern described in earlier chapters. Instances that need extra checks — because they have instance-specific flows — add them in a separate file rather than editing the template's file, so sync does not conflict with local work. This separation is the same principle as tooling-layer separation applied to testing: keep what the template owns in a path the template controls, and keep what the instance owns alongside it but never inside it.

There is a second reason to keep smoke tests template-owned. They are the canary that tells you whether a sync succeeded. If the template's smoke suite passed in the archetype and fails in the instance immediately after a sync, the failure is almost always in the sync, not in the test. Keeping both sides running the same test gives you that diagnostic signal for free.

## Validating Sync Did Not Break Anything

The third testing layer runs at the sync boundary, where upstream changes land in a downstream instance. Sync is where template systems fail silently. A file merges cleanly because git does not understand the semantics; a renamed function compiles because the old name is still exported; a config option is removed but the instance still references it from a path the archetype never sees. None of these show up as merge conflicts, and none of them show up in the archetype's tests. The only way to catch them is to run the instance's real test suite after the sync lands.

The pattern here is a sync-validation pipeline. When an agent or engineer opens a sync PR against a downstream instance, CI runs three things in sequence. First, the instance's existing test suite, including the inherited smoke tests from the previous section. This is the regression net: whatever was passing before the sync has to still pass after it. Second, a drift-check re-run, to confirm that the sync actually closed the drift it claimed to close. A sync PR that says "aligns with upstream v2.4.0" should produce a clean drift report against v2.4.0, and if it does not, something was missed or resolved incorrectly. Third, a build of the full downstream artifact — the site, the package, the container — because build-time failures catch a class of breakage that unit tests routinely miss. Linking errors, missing assets, and misconfigured paths all surface here.

Sync validation also needs a human-visible report. The drift-check script from Chapter 6 should produce output that a reviewer can read in under a minute: what files changed, which contracts were touched, whether any protected paths were modified. The goal is to let the reviewer approve or reject the sync based on the report, without having to re-derive what the sync actually did. When sync PRs are opened by an agent, this report is doubly important, because the agent's own judgment about which conflicts to resolve and how is part of what is under review.

Finally, a template ecosystem benefits from occasional cross-instance validation runs. A job that iterates over every known instance (from the template registry introduced in Chapter 10), runs its smoke suite against the latest archetype, and reports which instances would break if they synced today is worth the maintenance cost for any template with more than a handful of consumers. It turns sync breakage from a per-instance surprise into a per-release signal, and it lets the archetype's maintainers fix compatibility problems upstream rather than in fifteen separate downstream PRs. That is the payoff for taking template testing seriously: the template stays trustworthy not because nothing ever goes wrong, but because when something does, you find out before your consumers do.
