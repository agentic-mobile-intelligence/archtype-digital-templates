# Chapter 5: Tooling-Layer Separation

Pinning an upstream ref is only half of the discipline a healthy template ecosystem needs. The other half is structural: the template and the instance have to be *shaped* in a way that makes sync feasible in the first place. If build tooling and generated content are tangled together in the same directories, no amount of careful ref-pinning will save you — every upstream bump will create merge conflicts in the exact files downstream consumers are most likely to have touched. Tooling-layer separation is the architectural choice that keeps the upstream and the instance from fighting over the same bytes.

## Separating Build Tools from Content

Every template instance has two kinds of files. The first kind is **tooling** — build scripts, container definitions, CI workflows, validation utilities, outline parsers, generators. Tooling is code that *produces* something. Downstream consumers almost never need to edit it; they need it to *work*, and to keep working as it is upgraded. The second kind is **content** — the thing the tooling produces output from. For this book, content is the Markdown in `drafts/`, the metadata in `publishing/metadata.yaml`, and the outline in `publishing/outline.yaml`. Content is the reason the instance exists. It is edited constantly and diverges from upstream within hours of the fork.

The mistake that kills templates is letting these two kinds of files sit next to each other without a boundary. Consider a template where the build script expects `./chapter-01.md` and `./chapter-02.md` in the repository root, alongside `./build.sh` and `./Dockerfile`. The moment a downstream author adds a new chapter, renames one, or reorders them, they are editing the same directory the upstream will eventually want to update. When the upstream improves the build script, the downstream repo cannot pull that change without also pulling (or conflicting with) whatever chapter files upstream happens to ship as examples. The two layers are physically entangled, and every sync becomes a negotiation.

The fix is boring and effective: put tooling in one directory, content in another, and let each evolve on its own cadence. Tooling is owned by the upstream; downstream treats it as read-mostly. Content is owned by the instance; upstream treats it as a shape, not a payload. When the boundary is clean, "upgrade the tooling" becomes a single directory-scoped operation — overlay `publishing/` from the new upstream ref, run the tests, commit. No content files are touched. No negotiation.

Tooling-layer separation is also what makes the *minimal instance surface* from Chapter 3 physically realizable. If the instance's divergent surface is `drafts/` and a handful of config values, the rest of the repository can be synced mechanically. The separation is what turns "keep upstream and downstream aligned" from a hope into a procedure.

## Directory Conventions That Make Sync Easier

Separation needs conventions to be stable. A template that *mostly* separates tooling from content — but leaks a build helper into the content directory, or a draft into the tooling directory — will still generate merge conflicts, just less predictable ones. Three conventions keep the boundary sharp.

**One directory owns tooling; that directory is the unit of sync.** In this book's template, that directory is `publishing/`. Every build-related file lives under it: `build.sh`, `build-arxiv.sh`, `validate.sh`, `init.sh`, `stats.sh`, `Containerfile`, `latex-preamble.tex`, `generate-quick-reference.py`, and the YAML files that drive them. A sync script can say "replace the entire `publishing/` directory with the version at upstream ref X" and be confident that the operation will not clobber any authored content. The directory is the contract's unit; nothing smaller, nothing larger.

**Content directories never contain tooling, and tooling directories never contain content.** Drafts live in `drafts/`. Assets live in `assets/`. Concepts live in `concepts/`. None of them contain scripts, Containerfiles, or CI configuration. Conversely, `publishing/` contains no chapter Markdown, no author notes, no instance-specific data. Files like `publishing/metadata.yaml` and `publishing/outline.yaml` sit in a gray zone — they live in the tooling directory but carry instance-specific values — and the convention handles this explicitly: they are structured config, not free-form content, and they are expected to be edited per-instance while keeping the same *schema* as upstream. That schema stability is what lets the upstream ship tooling updates without breaking instance data.

**Reserved top-level paths stay reserved.** The top-level layout — `drafts/`, `publishing/`, `assets/`, `concepts/`, `archive/`, `book-cover/`, `.github/workflows/` — is part of the template's interface, not an accident of whoever set it up first. Downstream instances can add sibling directories for instance-specific needs, but they should not rename the reserved ones or nest them inside new wrappers. Renaming `drafts/` to `src/chapters/` in a single instance means every upstream build script improvement now requires a downstream translation layer. The shape of the directory tree is part of the contract.

The payoff of these conventions is that the sync operation becomes expressible in a single sentence: "bring `publishing/` and `.github/workflows/` up to the pinned upstream ref; leave everything else alone." When someone can describe the sync in one sentence, they can also automate it, review it, and trust it.

## The `publishing/` Directory as a Contract

Treat `publishing/` less as a folder and more as a *contract* between the upstream template and every downstream instance. The contract has three clauses.

The first clause is **behavioral**: given a repository that conforms to the template's content conventions, the scripts inside `publishing/` will produce a working build. An author running `publishing/build.sh` in a freshly forked instance expects the same outputs the upstream produces — a compiled manuscript, an arXiv-formatted variant from `publishing/build-arxiv.sh`, validation reports from `publishing/validate.sh`, quick-reference artifacts from `publishing/generate-quick-reference.py`. The upstream's job is to keep this guarantee true across versions; the downstream's job is to not break it by modifying the scripts locally.

The second clause is **structural**: the schema of the config files inside `publishing/` is stable within a major version. `publishing/outline.yaml` will always have a `book:` key, a `front_matter:` list, numbered `part_N:` sections with chapters, and `back_matter:` entries. `publishing/metadata.yaml` and `publishing/patterns.yaml` follow the same rule. Downstream instances fill in the values — title, subtitle, chapter titles, source files — but do not invent new top-level keys that upstream does not recognize. This is what makes instance-specific data survive an upstream sync: the field names the new tooling looks for are the field names the old tooling wrote.

The third clause is **ownership**: everything in `publishing/` that is *not* an instance-config file is owned by upstream. `build.sh`, `Containerfile`, `latex-preamble.tex`, `generate-quick-reference.py`, `run-container.sh`, `init.sh`, `stats.sh` — these are all upstream property. Downstream instances should resist the urge to patch them in place. If `build.sh` needs a new flag for an instance's use case, the right move is to backport the feature upstream (Chapter 7) or add an instance-level wrapper outside `publishing/` that calls into it. The contract is broken the moment a downstream edits a tool-owned file; from that point forward, every upstream bump either loses the edit or hits a merge conflict.

The directory-as-contract framing changes what "updating the template" means. It is no longer "pull a commit and resolve whatever conflicts appear." It is "replace the upstream-owned files under `publishing/` with the version at the pinned ref, re-run the build, and confirm the instance config still parses against the new schema." The three clauses make each of those steps verifiable: behavior has an oracle (the build either works or it does not), structure has a validator (`publishing/validate.sh`), and ownership has a clear rule (which files were touched?).

Separation, conventions, and contract are layered: separation creates the physical boundary, conventions keep the boundary sharp, and the contract defines what each side promises the other. With those three in place, the drift-detection tools of the next chapter have something real to check against, and the backport pattern of Chapter 7 has a clear inside and outside. Without them, every sync is a negotiation, and the template decays back into a one-time scaffold.
