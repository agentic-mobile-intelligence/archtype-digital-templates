# Chapter 3: Upstream and Downstream Defined

By now the shape of the problem is clear: the moment a template is forked, a second artifact exists, and the two begin to drift. Before we reach for tools and patterns to manage that drift, we need a shared vocabulary. The words *upstream* and *downstream* get used loosely in developer conversations — sometimes for Git remotes, sometimes for package dependencies, sometimes for data pipelines. When we talk about template ecosystems, they mean something specific, and getting the definitions right is the difference between a sync strategy that works and one that quietly decays.

This chapter establishes three working definitions: who owns the archetype, what lives on each side of the upstream/downstream boundary, and how to keep the instance small enough that drift stays manageable.

## Who Owns the Archetype

The *archetype* is the original template — the repository that defines the shape of every instance derived from it. Ownership of the archetype is the most important social question in a template ecosystem, and it is almost always answered badly at first.

The default pattern looks like this: a senior engineer scaffolds a project they are happy with, a teammate likes it and forks it, and soon there are four or five repositories that all look vaguely alike. Nobody is formally responsible for the archetype, because nobody ever said out loud that one existed. Each fork slowly adapts. When the original author improves their build pipeline, there is no mechanism to propagate that change, so the forks stagnate. When one of the forks discovers a better CI configuration, there is no mechanism to send that improvement back, so the archetype stagnates too.

Functional template ecosystems make ownership explicit. An archetype has a *maintainer* — an individual or team — whose job description includes accepting improvements, cutting releases, and communicating changes to consumers. The maintainer does not own every instance. The maintainer owns the *contract* that instances rely on.

A useful test: if you deleted the archetype today, would anyone notice within a week? If the answer is "no," then either the archetype is not actually in use, or its consumers have drifted so far that the archetype has become a historical artifact. Both outcomes mean ownership has lapsed. A healthy archetype has consumers who feel the absence of updates, a maintainer who feels the weight of requests, and a clear channel between them.

Ownership also implies boundaries. The archetype maintainer is not responsible for the business logic any downstream instance writes on top of the template. They are responsible for the parts every instance shares — the tooling, the directory conventions, the scripts, the configuration patterns. This distinction becomes the backbone of the next section.

## What Belongs Upstream vs Downstream

Once ownership is clear, the next question is where individual pieces of a repository live in the ecosystem. *Upstream* refers to the archetype — the canonical source. *Downstream* refers to any instance — any derived repository that started from the archetype and is now evolving on its own. Every file in an instance belongs to one of three categories:

**Upstream-owned.** These files exist to serve *every* instance of the template. They are authored in the archetype and should flow downward. Build scripts, lint configuration, CI workflows, publishing pipelines, and directory-layout conventions typically live here. The rule is: if a downstream repo edits one of these files, the edit is either (a) a local hack that should be removed or (b) a genuine improvement that should be backported. There is no third option where a downstream divergence is fine to leave in place indefinitely. That third option is how drift becomes permanent.

**Downstream-owned.** These files exist to serve *one specific instance*. In a content repository, this is the content itself — chapters, posts, products, whatever the instance actually produces. In an application repository, this is the domain code, the feature logic, the UI particular to this product. The archetype should not carry these files. If the archetype carries examples, it should carry *placeholders* that are obviously meant to be replaced, not real content that invites divergence through edits rather than deletion.

**Shared surface.** A narrow middle band exists where upstream defines the *shape* and downstream fills in the *values*. Configuration files are the canonical example: the archetype specifies that a `config.yaml` exists with certain keys, but the values are per-instance. Frontmatter schemas, environment variable templates, and manifest files all tend to live here. The shared surface is the trickiest region because both sides have a legitimate claim, and confusion about which side owns what is a leading cause of sync failures.

The design heuristic is that the shared surface should be *small*. Every file in this category is a negotiation point. The more negotiation points you create, the more places a sync can fail in surprising ways. Good archetypes push files firmly into upstream-owned or downstream-owned territory and keep the ambiguous middle narrow.

One consequence worth naming: the boundary is not always obvious from looking at the file tree. Two files sitting next to each other in the same directory can live on opposite sides of the line. A `README.md` might be upstream-owned (a template README explaining how to use the archetype) or downstream-owned (a project README describing the specific instance). The distinction is semantic, not structural. Making it explicit — often through a directory convention like `publishing/` or `template/` — is one of the highest-leverage decisions a template author can make. Chapter 5 returns to this in detail.

## The Principle of Minimal Instance Surface

The third definition follows from the first two, and it is the single most important principle in this book: **minimize the downstream surface.**

The *instance surface* is the set of files and changes a downstream instance contributes on top of its upstream archetype. If an instance touches ten files, its surface is ten files. If it reshapes the directory tree, overrides the build script, and rewrites the CI config, its surface is enormous — and the cost of every future sync grows proportionally. Drift is not a function of time alone. Drift is a function of surface area multiplied by time.

The principle has a few concrete implications.

First, **prefer configuration over modification.** When an instance needs behavior that differs from the archetype, the archetype should expose a configuration hook rather than forcing the instance to edit shared code. Every hook the archetype adds is a line the instance does not need to diverge on.

Second, **keep content clearly isolated.** A template should make it obvious which directories are the instance's domain — `drafts/`, `content/`, `src/app/`, whatever the convention is — so that a sync operation can safely ignore them. If content is interleaved with tooling, no sync strategy will ever feel safe.

Third, **treat every local override as a debt.** Downstream-only tweaks to upstream-owned files accumulate interest. Each one makes the next upstream pull harder. A discipline of asking *can this go upstream?* at the moment of writing a local change — rather than six months later when a sync breaks — is the single habit that distinguishes ecosystems that survive from ecosystems that fragment.

Minimal surface is not minimal *effort*. It takes work to keep instances small: identifying the right abstractions upstream, resisting the temptation to fork-and-forget, backporting improvements rather than hoarding them. But the payoff is that the ecosystem stays coherent. Every instance benefits from every improvement, every archetype change propagates without drama, and the template becomes what it was meant to be — a living foundation rather than a starting point that was immediately abandoned.

With these three definitions in place — ownership, boundary, and minimal surface — we are ready to look at the patterns that make them operational. Part 2 is about that machinery: how to pin, separate, detect, backport, and automate the relationship between upstream and downstream.
