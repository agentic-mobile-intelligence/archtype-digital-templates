# The Lifecycle of a Template Instance

Every template instance begins with an act of copying. You find a repository that looks close to what you need, you press a button, and suddenly you have your own version. The mechanics of that first step—how you copy, what gets carried over, what gets severed—determine the trajectory of everything that follows.

This chapter traces what happens from that initial moment through the first hours, days, and weeks of an instance's life. We will use `template-start-from-here` as our running example: a real template repository designed for book projects built with Pandoc, containing a `publishing/` directory with build tooling, a `drafts/` directory for content, and configuration files that tie the two together.

## Forking vs Cloning vs GitHub Template

There are three common ways to create a new project from an existing repository, and they are not interchangeable. Each one establishes a different relationship between your new project and the source.

**Forking** creates a new repository on GitHub that retains a permanent link back to the original. GitHub knows your fork's parentage. You can open pull requests against the upstream, and you can fetch upstream changes into your fork with a few commands. This sounds ideal for staying in sync, but forks carry baggage. Your fork inherits the entire commit history. GitHub's UI nudges you toward contributing back, which makes sense for open-source collaboration but creates noise when your instance is meant to diverge. If `template-start-from-here` has three years of commit history about decisions you were never part of, that history clutters your `git log` and makes it harder to understand what *your* project has done.

**Cloning** copies the repository to your local machine but does not create a new remote repository. You get the full history, the remote still points to the original, and you are one accidental `git push` away from writing to someone else's repository. Most people who clone a template immediately change the remote, at which point they have a disconnected copy with inherited history and no formal upstream relationship. It works, but it is manual and error-prone.

**GitHub's "Use this template" button** creates a new repository with the template's file tree but none of its commit history. Your new repo starts with a single commit. There is no fork relationship, no upstream link, no inherited history. This is the cleanest starting point for a project that intends to become its own thing—which is what most template instances are.

When you click "Use this template" on `template-start-from-here`, you get a fresh repository containing the `publishing/` directory, the `drafts/` stubs, the `Makefile`, and the configuration files. Your first commit is yours. Your history is yours. But the tradeoff is real: you have also severed every automated mechanism for knowing what has changed upstream since you copied it. There is no `git remote` pointing back. There is no "Sync fork" button. If the template's `Makefile` gets a critical bugfix next week, you will not hear about it unless you have built your own process for checking.

This tradeoff—clean start versus ongoing connection—is the central tension of template-driven development. The rest of this book explores how to manage it. For now, the important thing is to understand that your choice at creation time shapes the effort required for every future sync.

## The Divergence Clock Starts Immediately

The moment your instance exists, it begins to diverge from the template. This is not a defect. It is the entire point. You created an instance because you have a specific project to build, and that project is not identical to the template's assumptions.

With `template-start-from-here`, the very first thing you do is open `publishing/outline.yaml` and replace the placeholder structure with your actual book's chapters. You change the title in the metadata. You might rename draft files or add new ones. Within the first hour, you have already made changes that would conflict with the template's version of those same files.

This initial burst of customization is what I call the **divergence clock**. It starts the moment you create the instance, and it only moves in one direction. Every edit you make increases the distance between your project and the template it came from. Some of those edits are in files you were *meant* to change—content files, configuration, project-specific metadata. Others might touch files that the template author considers part of the stable tooling layer—the `Makefile`, CI configuration, build scripts.

The divergence clock has two important properties.

First, it accelerates. Early changes tend to be predictable: filling in content placeholders, adjusting configuration. But as the project matures, changes become more contextual. You add a custom build step because your project needs a glossary. You modify a script because your hosting environment differs from the template's assumption. Each customization makes the next upstream sync harder, because there are more places where your version and the template's version have made different choices.

Second, the clock does not distinguish between intentional and accidental divergence. When you deliberately customize `outline.yaml`, that is intentional—you meant to change it, and you know it differs from the template. But when you fix a typo in the `Makefile` or adjust a script's error handling, you may not think of that as divergence at all. You were just fixing a local problem. Six months later, when the template publishes a new version of that `Makefile`, you will discover that your "small fix" now creates a merge conflict. The divergence was real whether or not you were tracking it.

Understanding the divergence clock changes how you approach instance management. Instead of treating sync as something you will "deal with later," you begin to see it as something that gets harder with every passing commit. The question is not *whether* to sync but *how often* and *at what cost*.

## Mapping What Changes and Why

Not all divergence is equal. To manage an instance effectively, you need a mental model for categorizing changes. In the context of `template-start-from-here`, every file falls into one of three buckets.

**Content files** are the ones you are supposed to change. The draft Markdown files in `drafts/`, the outline in `publishing/outline.yaml`, any project-specific assets. These are the whole reason the instance exists. They will diverge from the template immediately and completely, and that is fine. Nobody expects to sync their chapter text back to a template.

**Tooling files** are the build infrastructure: the `Makefile`, scripts in `publishing/`, CI workflows, Pandoc configuration. The template author maintains these. They represent the template's opinion about how books should be built—how to compile Markdown to PDF, how to validate structure, how to manage draft status. In a healthy template relationship, you rarely touch these files. When the template improves its build pipeline, you want to pull those improvements without conflict.

**Configuration bridges** sit between content and tooling. They are files that the tooling reads but that you need to customize for your project. `outline.yaml` is the clearest example: the build system needs it to know which files to compile and in what order, but the actual content of the outline is project-specific. The `DRAFT_FILES` array in the `Makefile` is another—it is part of the build tooling, but its value depends on your specific chapters.

This three-bucket model reveals where template design succeeds and where it fails. A well-designed template minimizes configuration bridges by making them small and isolated. If all project-specific values live in a single configuration file, syncing the rest of the tooling is easy. A poorly designed template scatters project-specific values throughout the build system, turning every tooling file into a configuration bridge and making clean syncs nearly impossible.

When you are evaluating a template—or designing one—count the configuration bridges. Look at every file and ask: "If I pull a new version of this from the template, will it overwrite something I customized?" Files where the answer is yes are configuration bridges, and each one is a future merge conflict waiting to happen.

The lifecycle of a template instance, then, follows a predictable arc. You begin with a clean copy. You immediately customize the content files and configuration bridges. Over time, you may touch tooling files for project-specific reasons. Each of these changes increases your divergence from the template, and the effort required to sync grows accordingly.

The next chapter formalizes this relationship by defining what "upstream" and "downstream" mean in a template ecosystem, and introduces the principle that governs how much of the template's surface area your instance should expose: the Principle of Minimal Instance Surface.
