# Chapter 9: Versioning Your Template

Templates change. Build configurations get updated, directory structures get reorganized, CI pipelines get replaced. Every one of those changes ripples outward to every downstream instance that depends on the template. Without a versioning discipline, those ripples become invisible shockwaves — downstream teams discover breakage only after they attempt a sync and everything falls apart.

Versioning a template is not the same as versioning a library. Libraries have APIs with clear contracts. Templates have file trees, directory conventions, configuration defaults, and implicit assumptions baked into dozens of files. The challenge is not picking a version number — it is communicating what a version number actually means when your deliverable is a starting point for someone else's project.

This chapter lays out a practical versioning approach: how to apply semantic versioning to templates, how to write changelogs that downstream consumers can act on, and how to classify changes so that "breaking" and "non-breaking" have real, enforceable definitions.

## Semantic Versioning for Templates

Semantic versioning (semver) gives library authors a compact language for communicating the nature of a release: MAJOR.MINOR.PATCH. A major bump signals breaking changes. A minor bump adds functionality without breaking existing usage. A patch fixes bugs. The system works because libraries have a well-defined surface area — the public API — and consumers interact with it through function signatures, types, and documented behavior.

Templates do not have a public API in the traditional sense. But they do have a contract. The contract is the set of files, directory structures, configuration values, and conventions that downstream instances depend on. Versioning a template means defining that contract explicitly enough to apply semver to it.

Start by identifying your template's contract surface. This is the subset of the template that downstream instances are expected to keep in sync with. It typically includes:

- **Build and tooling configuration.** Makefiles, CI pipelines, Docker configurations, linting rules — the files that define how the project is built, tested, and deployed.
- **Directory structure conventions.** Where source code goes, where tests live, where configuration is expected. If your template assumes `src/` and a downstream instance reorganizes to `lib/`, syncing becomes a manual translation exercise.
- **Shared scripts and utilities.** Helper scripts that downstream instances are expected to use as-is, not customize.
- **Configuration defaults.** Values in config files that downstream instances may override but that the template provides as baselines.

Everything outside the contract surface is instance-specific content — the actual application code, business logic, and customizations that downstream teams own entirely. Changes to instance-specific content do not need versioning because they are not synced.

With the contract surface defined, semver maps naturally:

- **MAJOR** — a change that will break downstream instances if they sync without manual intervention. Renaming a directory that instances depend on, removing a CI step that instances reference, changing a configuration schema.
- **MINOR** — a change that adds new capability without altering existing contract surface. Adding a new CI job, introducing a new helper script, adding a new configuration option with a sensible default.
- **PATCH** — a bug fix or correction to existing contract surface that requires no downstream action. Fixing a typo in a CI config, correcting a broken script, updating a pinned dependency version.

Tag your template repository with these versions. Use annotated Git tags (`git tag -a v2.1.0 -m "Add pre-commit hook support"`) so the version history is part of the repository's permanent record. Downstream instances can then pin to a specific tag and upgrade deliberately, as described in Chapter 4's upstream pinning pattern.

One nuance that trips teams up: version zero. When a template is still finding its shape — directory structures are being reorganized, conventions are still being established — do not pretend stability exists. Use `0.x.y` versions and document that the contract surface is unstable. This honesty prevents downstream teams from building assumptions on top of a foundation that is still shifting.

## Changelogs Downstream Can Actually Use

A changelog is only useful if the person reading it can determine what they need to do. Most changelogs fail this test. They describe what changed from the author's perspective ("refactored build pipeline," "updated dependencies") without addressing what the change means for someone consuming the template.

A template changelog must answer three questions for every entry:

1. **What changed?** Be specific about files and structures, not abstractions. "Replaced `Makefile` with `justfile`" is actionable. "Improved build system" is not.
2. **Does it affect synced files?** If the change only touches files outside the contract surface, downstream instances can ignore it entirely. Say so explicitly.
3. **What does downstream need to do?** For breaking changes, provide a migration path. For non-breaking changes, describe the expected sync behavior — will a merge apply cleanly, or should the consumer expect conflicts in specific files?

Here is what a useful template changelog entry looks like:

```
## v3.0.0 — 2026-04-15

### Breaking

- **Replaced `Makefile` with `justfile` for build commands.**
  Affected files: `Makefile` (removed), `justfile` (added).
  Migration: Install just (https://github.com/casey/just). Rename any
  custom Make targets you added to justfile recipes. The default recipes
  mirror the old Make targets — run `just --list` to verify.

- **Moved CI config from `.github/workflows/ci.yml` to
  `.github/workflows/build.yml` and `.github/workflows/test.yml`.**
  Migration: Delete your existing `ci.yml`. The new split workflows
  will be added on sync. If you added custom steps to `ci.yml`,
  determine whether they belong in the build or test workflow and
  add them to the appropriate file.

### Added

- **New `scripts/lint-config.sh` for validating project configuration.**
  No downstream action required. File will be added on next sync.

### Fixed

- **Corrected Node.js version in `.nvmrc` from 18 to 20.**
  Syncs cleanly. No manual intervention needed unless you pinned
  a different version intentionally.
```

Notice the structure: each entry names the specific files involved, states whether downstream action is required, and when action is required, tells the consumer exactly what to do. This is more work for the template maintainer, but it is the difference between a changelog that gets read and one that gets ignored.

Keep the changelog in a `CHANGELOG.md` at the root of the template repository. Use the version numbers from your tags as the section headers. Downstream instances can diff the changelog between their current pinned version and the version they are upgrading to in order to build a complete migration checklist.

For teams with many downstream instances, consider supplementing the human-readable changelog with a machine-readable format — a JSON or YAML file that lists affected paths, change types, and migration flags. This enables the kind of automated sync tooling discussed in Chapter 8, where an agent can read the changelog and determine whether a sync can be applied automatically or requires human review.

## Breaking vs Non-Breaking Template Changes

The distinction between breaking and non-breaking changes is the most important judgment call in template versioning. Get it wrong in one direction and you flood downstream teams with unnecessary major version bumps that erode trust in the versioning scheme. Get it wrong in the other direction and a "minor" update silently breaks builds across dozens of instances.

The rule is straightforward: **a change is breaking if a downstream instance that syncs the change without reading the changelog will experience breakage.** Breakage means builds fail, tests fail, scripts error, or behavior changes in ways the consumer did not expect.

By that definition, here are changes that are almost always breaking:

- **Renaming or moving files in the contract surface.** If downstream instances reference `scripts/deploy.sh` and you rename it to `scripts/release.sh`, their automation breaks.
- **Removing files or configuration keys.** Anything that downstream instances might depend on — even if you think they should not — is a breaking removal.
- **Changing the schema of a configuration file.** If your template provides a `config.yaml` and you rename a top-level key, every instance that reads that key breaks.
- **Changing directory structure conventions.** If the template moves from `src/` to `pkg/`, instances that import from `src/` paths break.
- **Upgrading a major dependency version.** Bumping Node.js from 18 to 22, switching from Webpack to Vite, or replacing one test framework with another are all breaking changes even if the template itself still works — downstream instances may depend on the old tool's behavior.

And here are changes that are almost always non-breaking:

- **Adding new files that do not conflict with existing paths.** A new helper script, a new CI workflow with a unique name, a new configuration file.
- **Adding new configuration keys with defaults.** If existing behavior is preserved when the key is absent, the change is safe.
- **Updating pinned dependency patch versions.** Going from `1.2.3` to `1.2.7` within a minor version is almost always safe.
- **Fixing bugs in existing scripts.** If the script was not working correctly before, fixing it is a patch, not a breaking change — even if some downstream instance was accidentally depending on the broken behavior.

The gray area is where discipline matters. Some changes look non-breaking but have subtle downstream effects. Adding a new required field to a configuration file is breaking even though it is "just adding." Changing the default value of an existing configuration key is breaking if downstream instances rely on the old default without explicitly overriding it. Reordering steps in a CI pipeline is non-breaking in theory but breaking in practice if downstream instances inserted custom steps between specific existing steps.

When in doubt, call it breaking. A major version bump that turns out to be unnecessary is far less costly than a minor version bump that breaks production builds. Downstream teams can always read the changelog, confirm the change does not affect them, and upgrade quickly. But if you label a breaking change as minor, they will not read the changelog at all — they will just sync and discover the breakage the hard way.

One practical technique: maintain a `CONTRACT.md` file in your template repository that explicitly lists every file and convention that is part of the contract surface. When you make a change, check it against `CONTRACT.md`. If the change modifies anything listed there in a way that alters existing behavior, it is breaking. This makes the classification mechanical rather than a judgment call, which is exactly what you want when multiple people maintain the template.

Versioning a template is ultimately about respect for your downstream consumers' time. Every unversioned change forces them to reverse-engineer what happened. Every vague changelog entry forces them to read diffs and guess at intent. Every mislabeled breaking change forces them to debug failures they could have avoided. A disciplined versioning practice turns template updates from a source of friction into a predictable, manageable part of the development workflow.
