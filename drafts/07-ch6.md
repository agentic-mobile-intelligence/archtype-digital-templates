# Drift Detection

Every template instance begins its life as a faithful copy. The CI pipeline matches upstream. The linter config is identical. The directory layout follows the archetype exactly. Then someone adds a dependency. Someone else tweaks a build flag. A third person updates a config file that upstream also updated, but differently. Three months later, nobody can tell you how far the instance has drifted from its source — or whether syncing upstream changes will break everything.

Drift detection is the practice of measuring and surfacing that divergence before it becomes a crisis. This chapter covers what drift looks like in real template ecosystems, how to build scripts that quantify it, and how to wire those scripts into CI so that teams get alerts before drift becomes irreversible.

## What Drift Looks Like in Practice

Drift is not a single event. It is an accumulation of small, individually reasonable changes that collectively move an instance away from its archetype. Understanding the categories of drift helps you decide what to measure and what to ignore.

### Structural Drift

Structural drift occurs when the file tree itself diverges. The upstream template adds a new directory — say, `.github/workflows/` gains a `security-scan.yml` — and the instance never picks it up. Or the instance deletes a file that upstream still expects to exist. Structural drift is the easiest kind to detect because you can compare file lists mechanically, but it is also the most commonly ignored because missing files do not produce build errors until something depends on them.

A typical structural drift scenario: the archetype adds a `publishing/` directory contract (as discussed in Chapter 5) with a new validation script. Downstream instances that never pull this change will not fail immediately, but they will silently skip a validation step that upstream now considers mandatory. The longer this goes undetected, the more work accumulates when someone finally tries to sync.

### Content Drift

Content drift is subtler. The same file exists in both upstream and downstream, but the contents have diverged. Some content drift is intentional — the instance customized a configuration value that should stay local. But other content drift is accidental, the result of two independent edits to the same file that happened to touch different lines.

The challenge with content drift is distinguishing intentional customization from unintentional divergence. A downstream project that changed its `README.md` to describe its own product has not drifted in any meaningful sense. But a downstream project whose `.eslintrc` differs from upstream because someone fixed a lint rule locally — without sending that fix back upstream — has drifted in a way that matters.

### Dependency Drift

Dependency drift deserves its own category because it is the most dangerous. When an instance pins a different version of a shared dependency than what upstream specifies, the divergence may be invisible in day-to-day development but catastrophic during sync. Dependency drift compounds: if the instance is three major versions behind on a build tool, catching up requires migrating through every breaking change in sequence.

Dependency files — `package.json`, `Gemfile`, `go.mod`, `pyproject.toml` — are high-signal targets for drift detection. Even a single-line version difference in a lock file can indicate a significant divergence in the dependency tree.

### Configuration Drift

Configuration drift covers build configs, CI definitions, editor settings, and tool configurations. These files are particularly prone to drift because they are frequently edited to solve immediate problems ("the build broke, so I changed this flag") without considering whether the change should propagate upstream. Configuration drift is also where the most subtle bugs hide, because two projects can appear to build the same code while producing meaningfully different artifacts due to differing compiler flags, environment variables, or pipeline steps.

## Building a Drift-Check Script

A practical drift-check script does not need to be sophisticated. It needs to be runnable, deterministic, and clear about what it found. The goal is a report that a human or an automated system can act on, not a magical resolution engine.

### The Basic Approach

The simplest drift check compares the current instance against a known upstream ref. This requires that the instance tracks its upstream origin — a topic covered in Chapter 4's discussion of upstream pinning. Given a pinned ref, the script can perform three checks in sequence.

First, enumerate files present in upstream but missing from the instance, and files present in the instance but absent from upstream. This catches structural drift. Files that the instance is expected to add (like project-specific content) should be excluded via an allowlist.

Second, for files that exist in both locations, compute a diff. A simple approach is to compare checksums: if the SHA-256 of the upstream file matches the instance file, there is no content drift for that file. If they differ, the script records the file as drifted and optionally captures the diff for review.

Third, for known dependency files, perform a more targeted comparison. Rather than just flagging that `package.json` differs, parse both files and report which specific dependencies have version mismatches. This turns a vague "these files differ" into an actionable "you are three major versions behind on webpack."

### Structuring the Output

The drift-check script should produce machine-readable output — JSON or YAML — alongside a human-readable summary. A useful output structure includes a drift score (a simple count of drifted files or a weighted metric), a categorized list of drifted files, and a timestamp of the upstream ref used for comparison.

```yaml
drift_report:
  upstream_ref: "v2.4.0"
  instance: "project-alpha"
  checked_at: "2026-04-24T10:00:00Z"
  score: 7
  structural:
    missing_from_instance:
      - ".github/workflows/security-scan.yml"
    extra_in_instance:
      - "scripts/local-hack.sh"
  content:
    drifted_files:
      - path: ".eslintrc.json"
        upstream_sha: "a1b2c3..."
        instance_sha: "d4e5f6..."
      - path: "tsconfig.json"
        upstream_sha: "g7h8i9..."
        instance_sha: "j0k1l2..."
  dependencies:
    - file: "package.json"
      mismatches:
        - name: "webpack"
          upstream: "5.92.0"
          instance: "5.75.0"
        - name: "eslint"
          upstream: "9.4.0"
          instance: "8.56.0"
```

This structure lets automated systems parse the report and make decisions — for example, blocking a deploy if the drift score exceeds a threshold, or opening an issue if a critical file has diverged.

### Handling Intentional Differences

Not all drift is bad. The script must account for intentional divergence. The cleanest approach is an allowlist file — often called `.drift-ignore` or similar — that lives in the instance repository. This file lists paths or patterns that the instance has deliberately changed and that should not be flagged as drift.

```
# Instance-specific files, not tracked for drift
README.md
docs/**
src/config/local-overrides.ts
```

The allowlist serves double duty: it suppresses false positives in the drift report, and it documents which files the instance considers "owned" rather than "inherited." When a new team member asks why a particular config differs from upstream, the allowlist (or the absence of that file from the allowlist) provides the answer.

### Keeping the Script Simple

Resist the urge to build the drift-check script into a framework. A shell script or a single-file Python script that runs `git diff` against an upstream remote is sufficient for most teams. The important properties are that it runs without interactive input, exits with a nonzero code when drift exceeds a threshold, and produces output that can be piped to other tools. Complexity in the detection layer creates its own maintenance burden — and ironically, its own drift risk if the script itself diverges between instances.

## CI Integration for Drift Alerts

A drift-check script that lives in a developer's `~/bin` directory is a drift-check script that never runs. The real value comes from integrating drift detection into continuous integration, where it runs automatically and surfaces results where the team already looks.

### Choosing the Right Trigger

Drift checks should not run on every commit. They are comparing against an external reference (upstream), not validating local changes. The appropriate triggers are:

**Scheduled runs.** A weekly or biweekly cron job that runs the drift check and posts results to a team channel or dashboard. This provides a regular cadence of drift awareness without blocking any workflow.

**Upstream release events.** When the archetype publishes a new version, downstream instances should automatically run a drift check against the new ref. This tells the team exactly how much work a sync will require before they commit to it. If your template registry (Chapter 10) emits events on new releases, this trigger can be fully automated.

**Pre-sync checks.** Before attempting to merge upstream changes into an instance, run the drift check to produce a preview of what will conflict. This is not a gate — it is an informational step that helps the person performing the sync prepare for the merge.

### Surfacing Results

The drift report is only useful if the right people see it. Common integration points include:

**Pull request comments.** When a drift check runs as part of a PR pipeline, post the summary as a PR comment. This works especially well for pre-sync checks, where the PR itself represents a sync attempt and the drift report provides context for the reviewer.

**Dashboard or status page.** For organizations managing multiple instances from the same archetype, a central dashboard that aggregates drift scores across all instances provides a fleet-wide view. A simple static page generated from the JSON reports of each instance is enough — this does not require a dedicated application.

**Alerting thresholds.** Set a drift score above which the CI job posts to a team channel or creates a tracking issue. The exact threshold depends on context, but a reasonable starting point is to alert when more than 20 percent of tracked files have drifted or when any dependency is more than two major versions behind.

### Avoiding Alert Fatigue

The fastest way to kill a drift detection program is to make it noisy. If every CI run produces a wall of warnings about files that the team has intentionally diverged, developers will learn to ignore the output entirely. The allowlist discussed earlier is the first line of defense. Beyond that, categorize drift by severity. A missing optional workflow file is not the same as a two-major-version dependency gap. Weight the drift score accordingly, and only alert on high-severity items.

Consider a graduated response: informational reports run weekly and go to a dashboard; warnings trigger a message in the team channel; critical alerts — like security-related files drifting from upstream — create issues or block deploys. This tiered approach keeps drift visible without making it a constant interruption.

### Drift Detection as a Cultural Practice

Tooling alone does not solve drift. The scripts and CI jobs described in this chapter provide visibility, but acting on that visibility requires a team agreement that drift is worth managing. The most effective teams treat their drift report the way they treat test coverage: not as a metric to optimize to perfection, but as a signal that informs decisions. A rising drift score is not inherently bad — it may reflect intentional customization. But an unexamined rising drift score is a risk, because it means nobody is asking whether the divergence is intentional or accidental.
