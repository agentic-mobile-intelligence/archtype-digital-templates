# Chapter 7: The Backport Pattern

Every template ecosystem faces a recurring paradox. The developers working in downstream instances are closest to real-world problems — they encounter build failures, discover missing edge cases, and invent better ways of doing things. Yet those improvements stay locked inside a single instance, invisible to the upstream archetype and every other instance that would benefit from them. The backport pattern addresses this gap by creating a disciplined path for instance-level improvements to flow back upstream.

This chapter covers how to recognize when a downstream fix belongs in the archetype, how to structure your commits so they can travel cleanly between repositories, and how to run a backport PR workflow that keeps both upstream and downstream histories coherent.

## When Instance Improvements Belong Upstream

Not every change made in a downstream instance is a candidate for backporting. Most instance-level work is inherently local: populating content, adjusting configuration values, adding domain-specific business logic. These changes exist precisely because the instance has diverged from the archetype in intentional, expected ways. Backporting them would pollute the template with specifics it was designed to avoid.

The changes worth backporting share a common trait: they improve the machinery that every instance relies on, rather than the content that makes each instance unique. A few categories recur across template ecosystems.

**Bug fixes in shared tooling.** When a downstream developer discovers that the build script fails on a particular Node version, or that a CI workflow has a race condition, that fix applies to every instance. Leaving it downstream means every other consumer will eventually hit the same bug and solve it independently — or worse, never notice it until production.

**Infrastructure improvements.** A developer in one instance might add a Docker health check, improve the Makefile's clean target, or add a missing step to the release workflow. These changes touch the tooling layer that the archetype is responsible for. They belong upstream because they strengthen the foundation that all instances share.

**Documentation corrections.** If the template ships a README with setup instructions, and a downstream developer finds that a step is missing or wrong, that correction is a textbook backport candidate. The fix is universal, and every new instance created from the archetype will benefit immediately.

**Generalized patterns extracted from specific needs.** Sometimes a downstream developer builds something specific — say, a retry wrapper around an API client — and realizes the underlying pattern (a configurable retry utility) would be useful in the archetype itself. This category requires the most judgment. The backport is only appropriate if the extraction produces something genuinely general-purpose, not a thinly disguised version of the instance's domain logic.

A useful heuristic: ask whether the change would make sense to a developer who has just created a brand-new instance from the archetype and has never seen your project. If it would confuse them or require context they do not have, it probably does not belong upstream. If it would save them time or prevent a problem they would otherwise face, it is a backport candidate.

There is also a cultural dimension. Teams that backport regularly build a feedback loop between instance work and archetype maintenance. Developers start to notice upstream-worthy improvements as they make them, rather than only in retrospect. This awareness is one of the most valuable side effects of adopting the backport pattern — it trains the team to think in terms of layers.

## Structuring Commits for Backportability

The single biggest obstacle to backporting is entangled commits. When a developer fixes a build script bug in the same commit that adds a project-specific feature, extracting the fix for upstream becomes a surgical operation: cherry-picking, manually editing hunks, resolving conflicts against a different file tree. Most teams will not bother, and the fix stays downstream forever.

Structuring commits for backportability means making the upstream-worthy change easy to isolate after the fact — or better yet, isolating it at the time it is made.

**Separate concerns into distinct commits.** This is the foundational rule. If you are fixing a bug in the CI workflow and also adding a new project-specific job to that same workflow file, make two commits. The first commit contains only the bug fix. The second commit adds the project-specific job. The first commit can be cherry-picked cleanly; the second never needs to leave the instance.

**Keep upstream-worthy commits self-contained.** A backportable commit should not depend on changes introduced in surrounding instance-specific commits. It should apply cleanly to the archetype repository at the ref the instance was last synced from. This means avoiding references to instance-specific paths, variables, or configuration values within the commit.

**Use conventional scoping to signal intent.** Many teams adopt a prefix convention to mark commits that are candidates for backporting. A common approach is to prefix the commit message with `[upstream]` or use a conventional-commit scope like `fix(template):`. This makes backport candidates easy to find with `git log --grep` and creates a clear audit trail of what has been contributed back.

```
# Easy to find later
git log --grep='\[upstream\]' --oneline

# Example commits
[upstream] fix: correct Node version matrix in CI workflow
[upstream] docs: add missing Docker setup step to README
feat(project): add customer dashboard routes
```

**Avoid reformatting in backport commits.** If the archetype uses different formatting rules than your instance — different indentation, different line lengths, a different linter configuration — do not include formatting changes in a backportable commit. The commit should match the archetype's style, not the instance's. This often means making the fix, committing it against the upstream style, and then applying any instance-specific formatting in a separate commit.

**Write commit messages that stand alone.** A commit message destined for upstream should make sense without the context of your instance's history. "Fix the bug Sarah reported" is useless in the archetype. "Fix race condition in parallel test runner when TMPDIR contains spaces" tells every downstream consumer exactly what changed and why. Include the reasoning, not just the action.

These practices have a cost: they require more deliberate commit hygiene than most developers practice by default. The return is that backporting becomes a low-friction operation instead of an archaeological expedition. Over time, the discipline also improves the instance's own commit history, making it easier to review, bisect, and understand.

## The Backport PR Workflow

With well-structured commits in hand, the mechanical process of backporting follows a predictable workflow. The specifics vary depending on whether the archetype and instance share Git history (fork-based) or are fully separate repositories (template-based), but the core steps are the same.

### Step 1: Identify the Commits

Start by finding the commits that are candidates for backporting. If you have been using a commit prefix convention, this is a single command:

```bash
git log --grep='\[upstream\]' --oneline origin/main..HEAD
```

This shows all upstream-tagged commits on your instance's main branch that have not yet been merged into the archetype. Review this list and confirm that each commit is still relevant — some fixes may have already been independently addressed upstream.

### Step 2: Prepare the Backport Branch

Create a branch in the archetype repository (or your fork of it) based on the current upstream HEAD:

```bash
# In the archetype repository
git checkout -b backport/fix-ci-race-condition upstream/main
```

If you are working in a fork-based setup where the instance and archetype share history, you can cherry-pick directly. If the repositories are separate, you will need to apply the changes manually or use `git format-patch` and `git am` to transfer commits between repositories:

```bash
# In the instance repository: export the commit as a patch
git format-patch -1 abc1234 --stdout > /tmp/fix-ci-race.patch

# In the archetype repository: apply the patch
git am /tmp/fix-ci-race.patch
```

The `format-patch` / `am` approach preserves the original commit message and authorship, which matters for attribution and for maintaining a clear record of where the fix originated.

### Step 3: Verify Against the Archetype

Before opening a pull request, verify that the backported change works in the archetype's context. Run the archetype's test suite, build pipeline, and any validation scripts. A fix that works in your instance may fail upstream if the archetype has diverged since you last synced — different dependency versions, different directory structures, different CI configurations.

This verification step is where many backport attempts fail, and it is also where the value of regular upstream syncing (covered in Chapters 4 and 6) becomes apparent. The less drift between your instance and the archetype, the more likely a cherry-pick applies cleanly.

### Step 4: Open the PR with Context

Open a pull request against the archetype repository. The PR description should include:

- **What** the change does, in terms the archetype maintainers will understand.
- **Where** the fix originated — which instance encountered the problem.
- **Why** it belongs upstream — the argument that this is a template-layer concern, not an instance-specific one.
- **How** to verify — steps or tests that confirm the fix works in the archetype context.

```markdown
## Backport: Fix race condition in parallel test runner

**Origin:** acme-web-app instance (encountered in CI on 2026-04-15)

**Problem:** When TMPDIR contains spaces, the parallel test runner
script splits paths incorrectly, causing intermittent failures.

**Why upstream:** The test runner script ships with the archetype
and is used by all instances. Any instance running on a system with
spaces in TMPDIR will hit this bug.

**Verification:** Added a test case with a space-containing TMPDIR.
All existing tests continue to pass.
```

### Step 5: Close the Loop

Once the backport PR is merged upstream, update your instance to incorporate the change through the normal upstream sync process rather than keeping your local version. This prevents a future sync from creating a merge conflict between your original fix and the upstream version of the same fix.

If your team tracks backport status, mark the original instance commit as "backported" — whether through a label, a comment, or a metadata file. This prevents duplicate backport attempts and gives visibility into the flow of improvements between instance and archetype.

### When Backports Get Rejected

Not every backport PR will be accepted, and that is fine. The archetype maintainers may decide that a fix is too instance-specific, that the approach does not match upstream conventions, or that the problem should be solved differently at the template level. When this happens, keep the fix in your instance and move on. The commit structure you used to isolate the fix still benefits your own repository, and the PR discussion often surfaces useful context about the archetype's design philosophy.

---

The backport pattern is not just a mechanical workflow — it is a mindset shift. It asks downstream developers to see themselves as contributors to the ecosystem, not just consumers of it. When practiced consistently, it transforms the relationship between archetype and instance from a one-way broadcast into a collaborative loop. Fixes flow upstream, improvements propagate to all instances, and the archetype itself gets better because it is informed by real-world usage rather than theoretical design. The next chapter explores how agentic systems can automate parts of this loop, reducing the manual overhead that often prevents teams from backporting as frequently as they should.
