# Chapter 10: Multi-Instance Governance

A template serving one project is easy. You maintain the archetype, you maintain the instance, and the feedback loop is tight. But templates succeed precisely because they get reused, and the moment a second team forks your archetype, you have a governance problem. By the time twenty teams have done it, you have an ecosystem—whether you planned for one or not.

This chapter addresses the questions that surface when your template has many consumers: how you track what exists, how you communicate changes, and how you build infrastructure to manage the relationship at scale.

## Tracking Which Instances Exist

The first governance challenge is deceptively simple: do you know who is using your template? In most organizations, the answer is no. GitHub's template repository feature creates a new repo with no ongoing link to the source. Forks show up in the network graph, but teams that clone and push to a new remote vanish entirely. The result is a population of instances you cannot enumerate.

This matters because governance without visibility is fiction. You cannot communicate breaking changes to consumers you do not know about. You cannot measure drift across instances you have never cataloged. You cannot justify investment in template quality when you have no data on adoption.

### Discovery Approaches

Several strategies exist for tracking instances, each trading accuracy against friction.

**Marker files** are the simplest approach: embed a machine-readable file in the template itself—a `.template-origin.json` or a field in an existing manifest—that records the archetype name, the origin commit, and an instance identifier. Because the file ships with every new instance, you can scan your organization's repositories for it. The downside is that consumers can delete it, and discovery depends on broad read access across your SCM platform.

**Registration at creation time** is more reliable. If your template ships with a setup script or an `init` command, that script can POST to a central registry. This captures the instance at birth, but cannot prevent unregistered forks—someone will always `git clone` and skip the script.

**CI-based phone-home** is a middle path. Embed a lightweight telemetry step in the template's CI pipeline that reports the repository name, current template version, and drift status to a central endpoint. Any instance running CI will eventually appear in your registry, and you get ongoing liveness data rather than a single registration point.

**Periodic scanning** is the most passive approach. If you control the SCM platform, run scheduled scans looking for marker files or characteristic fingerprints—directory structures, config files, signature dependencies. It requires no cooperation from consumers but only works within platforms you administer.

In practice, the most robust tracking combines marker files with one active mechanism. Embed the marker so passive discovery is always possible, then layer on CI reporting or registration for real-time data.

### What to Track

Once you can enumerate instances, decide what metadata to collect. At minimum:

- **Instance identity.** The repository URL and the team that owns it.
- **Origin version.** The template version or commit the instance was created from.
- **Current version.** The version the instance most recently synced to.
- **Last sync date.** When the instance last pulled upstream changes.
- **Drift status.** Whether tracked files have diverged, and by how much.

This metadata turns governance from guesswork into something you can act on. You can identify which instances are dangerously behind, which teams never sync, and which parts of the template cause the most downstream divergence.

## Communicating Template Changes to Consumers

Knowing who your consumers are is only half the problem. The other half is telling them when something changes, in a way they can act on without drowning in noise.

### The Notification Spectrum

Change communication runs from passive to active.

At the **passive** end, you publish a changelog and expect consumers to check it. This works for small, engaged communities but fails as instance counts grow. No one checks a changelog they do not know exists.

In the **middle**, automated notifications fire when a new version is tagged, creating an issue or chat message in each consumer's channel. This is better because information goes to the consumer rather than waiting to be sought. The risk is notification fatigue—if every minor tweak generates an alert, teams mute them.

At the **active** end, an automated process opens pull requests directly against consumer repositories, pre-populated with the sync changes. The consumer sees exactly what will change and can merge, modify, or decline. This is the most actionable format and the most expensive to maintain.

### Segmenting by Impact

Avoiding fatigue requires segmenting changes by their impact on consumers.

**Breaking changes** affect files consumers must keep in sync—a new required environment variable, a changed build target, a restructured CI pipeline. These need active notification, ideally with a migration guide. Semantic versioning (Chapter 9) provides the signal; governance provides the delivery.

**Non-breaking improvements** update files consumers may or may not have modified—a new lint rule, a dependency bump, an improved Makefile target. These warrant a notification but not an urgent one. A periodic digest works better than individual alerts.

**Internal-only changes** modify template tooling, docs, or testing infrastructure that does not affect instances. These should generate no consumer-facing notification. The ability to make this distinction depends on the tooling-layer separation from Chapter 5: cleanly separated internals let you filter notifications by which layer changed.

### Channels

Where you send notifications matters as much as what you send. GitHub issues or PRs meet developers where they work and integrate with triage. Slack is good for awareness but bad for tracking—messages get buried. Email digests work for periodic summaries but are too slow for breaking changes. A dashboard gives a pull-based option for teams that prefer to check on their own schedule. The best systems layer multiple channels: automated PRs for breaking changes, chat for new releases, a dashboard for on-demand status.

## The Template Registry Pattern

Tracking instances, collecting metadata, and segmenting notifications converge on a single architectural pattern: the template registry.

A template registry is a service—or even a structured data file in a well-known location—that acts as the central coordination point between a template and its instances. It is not a package manager, though it borrows concepts. It is specifically designed for the template lifecycle: creation, tracking, drift detection, and change communication.

### Registry Components

A minimal registry has four components.

**A catalog of templates.** Each entry records the template name, its repository URL, current version, and owner. In organizations with multiple templates, this provides a single place to discover what archetypes exist.

**An instance ledger.** Each entry records an instance's identity, its origin template, current sync state, and owner. This is the structure that answers "who is using this template and how current are they?"

**A change feed.** When a template is updated, the registry records the new version, change category (breaking, non-breaking, internal), and affected files. Consumers subscribe or poll; automated systems use it to trigger notifications or sync PRs.

**A drift summary.** Periodically, the registry computes and stores drift data for each instance: which tracked files have diverged, by how much, and when they last matched. This powers dashboards and alerts.

### Implementation Options

The pattern can be implemented at several levels.

**File-based registry.** The simplest form is a YAML or JSON file in a well-known repository. The catalog is a list of entries. The ledger is populated by CI pipelines that append or update their entry on each run. The change feed is the template's own changelog, structured for machine readability. Drift summaries come from a scheduled job that clones each instance and compares tracked files. No infrastructure required beyond a Git host and CI.

**API-backed registry.** A small web service wraps the data. Registration becomes an API call. The change feed becomes a webhook publisher. Drift summaries are computed on demand or cached. The API can enforce policies—blocking instance creation without registration, or requiring acknowledgment of breaking changes before a deadline.

**Platform-integrated registry.** Integration with an internal developer portal handles registration automatically at instance creation. Drift data feeds existing observability dashboards. Change notifications route through the organization's standard alerting infrastructure. The most seamless experience, and the most platform engineering investment.

### Making Governance Lightweight

Regardless of implementation, the registry succeeds or fails on one criterion: does it feel lightweight rather than burdensome?

If registering an instance requires a form with twenty fields, teams skip it. If every template change triggers a Slack storm, teams mute the channel. If the drift dashboard requires a PhD to interpret, no one looks at it.

The best registries are nearly invisible during normal operation. Registration happens automatically as part of instance creation. Notifications arrive only when action is required and include clear instructions. Drift data reduces to a simple status—green, yellow, red—with details available on demand.

Governance at scale is not about control. It is about maintaining the feedback loop between template authors and template consumers so that the ecosystem can evolve together rather than silently diverging until the template becomes irrelevant. The registry pattern provides the infrastructure for that feedback loop. The culture of using it—of treating instances as ongoing participants rather than one-time copies—is what makes multi-instance governance actually work.
