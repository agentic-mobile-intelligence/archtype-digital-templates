# Chapter 12: Agentic Template Generation

Earlier chapters treated templates as artifacts that humans create, fork, customize, and synchronize. As agentic systems become standard parts of the development workflow, a new question emerges: what happens when an agent is the one generating the instance, customizing it for a specific context, and keeping it aligned with the upstream archetype over time?

This chapter explores that frontier. We move from agents as assistants in the sync process (Chapter 8) to agents as primary actors in the template lifecycle itself.

## Generating Template Instances with Agents

When a human developer forks a template, they bring context that is difficult to articulate: team naming conventions, deployment target, which optional features they need, and which ones will just add noise. A static template repository cannot capture that context. It offers a fixed starting point and leaves customization as an exercise for the reader.

An agentic generation workflow inverts the arrangement. Instead of forking a snapshot and manually editing it, the developer describes what they need, and an agent produces a tailored instance from the archetype.

The simplest version is a prompted scaffolding step. The agent reads the template repository and a specification—perhaps a YAML file describing project name, language variant, CI provider, and deployment target—and produces a new repository reflecting those choices. This is not fundamentally different from Cookiecutter or Yeoman in effect, except in one critical respect: the agent can reason about dependencies between choices. If you select PostgreSQL, the agent does not just swap a placeholder string. It adjusts the Docker Compose configuration, selects an appropriate ORM, updates CI to include a database service, and modifies the README. These cross-cutting concerns are exactly what static template variables handle poorly.

The architecture for agentic generation typically follows three phases. **Ingestion**: the agent reads the archetype and builds a model of its structure—which files are parameterized, which are static, and which contain conditional blocks. The agent must understand the template not as a flat file collection but as a graph of dependencies, where CI configuration depends on language choice, which depends on framework choice, which constrains the testing setup. **Resolution**: the agent takes the user's requirements and resolves them against the template's option space. A specification might say "deploy to AWS Lambda" without mentioning the implied bundling strategy, IAM role structure, and dependency constraints. The agent resolves these implications and asks for clarification only when genuinely ambiguous. **Emission**: the agent produces the output repository. Every file is written in its final form—no template markers, no TODO comments where a human was supposed to fill in a value.

The key insight is that the agent's output is still an instance of the archetype. It is not a novel creation. All the upstream-downstream machinery from earlier chapters still applies. The agent should record which archetype version it generated from, which specification it used, and which choices it made. This metadata becomes critical when the archetype updates.

## Agent-Driven Customization at Fork Time

Generation and customization are not the same. Generation produces a working instance from a specification. Customization adapts that instance to fit context the specification did not fully capture.

Consider a platform team that maintains an archetype for internal microservices. The archetype defines the standard project structure, approved dependency set, CI pipeline, and observability configuration. When a product team needs a new service, an agent generates an instance from a short specification. But the product team also needs customizations that do not fit neatly into a spec field: custom middleware for their auth scheme, an additional migration for a shared schema, a feature flag provider only their division uses.

Agent-driven customization handles this by treating the freshly generated instance as the starting point for a second pass. The agent takes the generated instance plus a set of customization instructions—potentially in natural language—and applies them. The critical constraint is that customizations must preserve the instance's relationship to the archetype.

This means the agent must distinguish between three categories of changes. **Archetype-owned files** are controlled entirely upstream—the CI pipeline, linting rules, base Docker image. The agent should not customize these directly. If a customization requires a change to an archetype-owned file, the agent should create an override layer: a local file that extends or patches the upstream one. **Instance-owned files** belong entirely to the downstream project—application code, business logic, domain tests. The agent can modify these freely because they will never conflict with upstream changes. **Shared files** contain both archetype-controlled sections and instance-specific content. The `README.md` is a classic example. For shared files, the agent should use the marker patterns from Chapter 5—clearly delineated sections indicating which parts are upstream-managed and which are instance-managed.

The practical implication is that the agent needs a model of file ownership, best expressed as a manifest in the archetype: a file declaring, for each path or pattern, whether it is archetype-owned, instance-owned, or shared. Without this manifest, the agent must infer ownership from heuristics, which leads to the same category of bugs that manual customization produces.

When the agent completes customization, it should produce a diff summary that separates archetype-compatible changes from changes that may create future sync friction. This summary becomes part of the instance's documentation, giving future maintainers a map of where the instance has diverged and why.

## Keeping Agents Synchronized with Template Versions

Generation is a point-in-time event. The harder problem is what happens when the archetype evolves and the instance needs to follow.

In a manual workflow, a developer reviews the archetype's changelog, identifies relevant changes, and applies them. This process is slow, error-prone, and frequently skipped. The drift detection and sync patterns from Chapters 6 and 8 address the mechanical aspects, but agentic generation adds a new dimension: the agent that generated the instance can also be the agent that keeps it synchronized.

This works because the agent has context a generic sync tool lacks. It knows the original specification. It knows which customizations were applied and why. When an archetype update changes a file the agent customized during generation, the agent can reason about whether the customization is still valid, needs adaptation, or should be discarded in favor of the new upstream version.

The synchronization workflow has three components. **Version tracking**: the instance must record which archetype version it was generated from, which agent version performed the generation, and which specification was used. All three can change independently, and each can trigger a different kind of sync. **Change classification**: when an archetype update arrives, the sync agent classifies each changed file against the ownership manifest. Archetype-owned files are updated directly. Instance-owned files are left alone. Shared files require merge logic—the agent examines the upstream diff and the instance's current state, then produces a merged version that preserves instance customizations while incorporating upstream changes. **Regression validation**: after applying changes, the agent runs the instance's test suite, builds the project, and verifies that the deployment configuration is still valid. If any check fails, the agent does not silently commit the broken state. It produces a detailed report and flags issues for human review.

The most important architectural decision is where the sync agent's authority ends. A fully autonomous agent that can merge upstream changes, resolve conflicts, and push to production without human review is technically possible but operationally dangerous. The better pattern is what Chapter 8 called human-in-the-loop approval gates, applied specifically to the sync context.

The agent should be authorized to perform the full sync workflow—detect changes, classify them, produce merged files, run validation—and then present the result as a pull request. The PR description should include the archetype version being synced to, the list of files changed with their ownership classification, any conflicts that required judgment calls, and the validation results. A human reviewer then approves or requests changes.

Over time, as trust builds, approval gates can be relaxed selectively. Archetype-owned file updates that pass validation might be auto-merged. But instance-owned files that the agent wants to modify should always require human review, because a change there in response to an archetype update suggests either a misclassification in the ownership manifest or a deeper architectural issue.

The end state is a template ecosystem where archetypes evolve continuously, instances are generated on demand with full customization, and synchronization happens as a regular, low-friction operation rather than a dreaded quarterly migration. The agent is not replacing the developer's judgment. It is removing the mechanical burden that causes developers to skip synchronization entirely, which is the real source of template drift.
