# Chapter 8: Agentic Sync Agents

The previous two chapters described the two halves of a sustainable template loop. Drift detection tells you where an instance has fallen out of step with its upstream. The backport pattern tells you how to move an improvement the other direction. Both are repeatable. Both are boring. And both are performed, today, in most organizations, by a human who opens a terminal, runs a script, reads a diff, writes a commit message, and clicks through a pull request interface.

That human is doing work an agent can do. Not all of it — the judgment calls are still human — but the mechanical portion, which is most of it, can be delegated. This chapter is about that delegation: how to wrap the drift and backport workflows inside agents that run on a schedule, propose changes as pull requests, and hand the final decision back to a person.

The goal is not to remove humans from the loop. It is to remove them from the parts of the loop where their attention adds nothing and to concentrate their attention where it matters.

## Automating Drift Detection with an Agent

The drift-check script from Chapter 6 is already most of the way there. It is a deterministic program that takes a downstream instance, compares it against a known upstream ref, and emits a structured report of which tracked files have diverged. An agent's job is to run that program on a cadence, interpret the output, and decide what to do next.

The cadence is a design choice with real consequences. Running drift checks on every push is noisy and expensive; running them monthly means drift accumulates into a cliff that nobody wants to climb. A weekly schedule is a reasonable default for most template ecosystems. Fast-moving upstreams may warrant daily runs; slow, stable ones can drop to biweekly. The important thing is that the schedule is predictable, because it changes the social contract: downstream maintainers start to expect the weekly drift report the way they expect the weekly dependency-update digest.

A sync agent in its simplest form is a job that runs on schedule, invokes the drift script, and posts the result somewhere a human will see it. That "somewhere" matters. A Slack channel is low-friction but easy to tune out. A GitHub issue filed on the downstream repository is harder to ignore and has the right lifecycle — open while drift exists, closed when it is resolved. An issue per drift event is too noisy; a single long-lived "drift status" issue that the agent updates in place is usually right.

The agent should also be careful about what counts as drift. Not all diffs are meaningful. A file whose only difference is a timestamp, a generated lockfile hash, or a rendered output is noise. The drift-check script should already exclude those, but the agent is the right place to apply a second pass of judgment: if the only drift is in files the instance is explicitly allowed to diverge on, the agent should say so and suppress the alert rather than creating busywork for the downstream team.

Above simple reporting, a slightly more ambitious agent classifies drift by direction. Upstream-ahead drift (the archetype has changed, the instance has not) is a candidate for a sync PR. Downstream-ahead drift (the instance has changed, the archetype has not) is a candidate for a backport PR. Both-directions drift is the interesting case, because it usually means a conflict that will require a human to reconcile. The agent does not need to solve that reconciliation — it just needs to name it and surface it.

## Proposing Sync PRs Automatically

Reporting drift is useful. Proposing the fix is transformative. The step from "your instance is behind upstream on these files" to "here is a pull request that pulls the upstream changes in, with the test suite green" is the step that turns a sync agent from a dashboard into a collaborator.

The mechanical path is straightforward. For each file the drift report flags as upstream-ahead, the agent checks out a new branch on the downstream repository, copies the upstream version of the file in, runs the instance's test and lint commands, and opens a pull request. If the tests pass, the PR is marked ready for review. If they fail, the PR is opened as a draft with the failure output in the body, and the agent tags whoever owns that area of the code.

Two details separate a useful sync PR from an annoying one. The first is scope. A PR that bundles thirty unrelated upstream changes is unreviewable; a PR per file is spammy. The right unit is usually a thematic grouping: all the CI changes together, all the linter-config changes together, all the build-tooling changes together. The directory conventions from Chapter 5 make this grouping obvious — if the template separates `publishing/`, `.github/`, and content directories cleanly, the agent can use those directories as natural PR boundaries.

The second detail is provenance. Every sync PR should state, in its body, what upstream commit range it is importing and why. A body that says "syncing `publishing/` from archetype@a1b2c3d..e4f5g6h — see upstream changelog entries X, Y, Z" gives the reviewer everything they need to decide whether to merge. A body that just says "sync" gives them nothing and forces them to reconstruct the context by hand. The agent should always pull the upstream commit messages into the PR body; they are free context the reviewer will appreciate.

Backport PRs are the mirror image and follow the same shape. When the agent detects that an instance has made a change to a tracked file that is not present upstream, and the change looks generalizable (small, self-contained, not obviously instance-specific), it opens a PR against the upstream archetype repository with the commit cherry-picked across. The upstream maintainer then decides whether to adopt it. Most backport candidates will be rejected, and that is fine — the cost of proposing is low, and the occasional accepted backport prevents the instance from drifting further.

A well-tuned sync agent will produce, over time, a steady trickle of small PRs in both directions. The trickle is the point. Drift that arrives continuously in reviewable chunks is drift that never becomes a project.

## Human-in-the-Loop Approval Gates

An agent that opens pull requests but cannot merge them is, by construction, safe. The pull request interface is itself the approval gate. A human has to click the button, and that click is the place where judgment enters the system.

This is worth stating plainly because there is a temptation, once an agent reliably produces green sync PRs, to let it auto-merge them. Resist this temptation for template sync work specifically. The reason is that template changes are cross-cutting: a change that looks safe in the upstream archetype (a CI-config tweak, a new linter rule, a bumped base image) can interact badly with something instance-specific that the agent has no visibility into. The test suite catches functional regressions. It does not catch "this change is technically correct but our team was in the middle of migrating away from that tool and this undoes three days of work." Only a human on the downstream team knows that context, and the PR review is the only place they can apply it.

That said, there are gradations between "agent proposes, human reviews everything" and "agent auto-merges everything." A practical middle path is to let the agent auto-merge a narrow category of changes — typically upstream updates to files the downstream team has explicitly declared as fully delegated, like `.github/workflows/` content that is always tracked verbatim. For every other file, the PR waits for a human. The delegation list should be short, explicit, and easy to audit: a `.template-sync.yml` or similar config in the downstream repo that names the paths the agent is allowed to auto-merge. Anything not on the list requires review. Anything that touches the list itself, of course, requires review.

Approval gates also need to work in the failure direction. When the agent's own checks fail — the test suite breaks, a conflict cannot be auto-resolved, the upstream ref has been force-pushed — the agent should not silently give up. It should surface the failure as a task a human can pick up, with enough context in the issue body that the human does not have to reproduce the investigation. "Sync from upstream@e4f5g6h failed: merge conflict in `publishing/scripts/build.sh`, lines 42-58" is actionable. "Sync failed" is not.

The final approval gate worth naming is the one on the agent itself. Any change to what the sync agent does — a new cadence, a new auto-merge path, a new PR template — is a change to how the template ecosystem operates, and should be reviewed like any other infrastructure change. The agent is automation, and automation that edits itself without supervision is how template ecosystems drift in ways that are much harder to detect than the file-level drift the agent was built to catch.
